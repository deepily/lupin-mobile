# Tier 1 Plan — Authentication + WebSocket Session Persistence

**Status**: ✅ Code complete — awaiting on-device verification
**Parent doc**: [Resync Audit](2026.04.15-resync-mobile-with-lupin-api-v0.1.6.md)
**Tier scope (from audit)**: Foundation tier — blocks every other tier.

---

## Context

The mobile app currently uses a hardcoded mock token (`Bearer mock_token_email_{userId}`) for all WebSocket auth and has no real user identity. Backend v0.1.6 exposes a full JWT auth system at `/auth/*` (10 endpoints). Additionally the WebSocket session ID (the "wise penguin" adjective-noun) is fetched fresh from `/api/get-session-id` on every app launch — losing in-progress conversations, queue subscriptions, and notification history continuity.

Tier 1 makes the mobile app a real authenticated client.

## Scope

### A. Authentication (login-only, single user)
Endpoints to wire (4 of 10 in Tier 1):
- `POST /auth/login` → returns access + refresh tokens
- `POST /auth/refresh`
- `POST /auth/logout`
- `GET  /auth/me`

UI surface:
- Login screen (already scaffolded in `lib/features/auth/`) — pre-populated with last-used email
- Logout from settings
- (No registration, password-reset, change-password, or email-verification UI in Tier 1)

Token lifecycle:
- Secure storage (`flutter_secure_storage`) for refresh token + last-used email
- In-memory access token with auto-refresh on 401
- Auth interceptor on the central HTTP client (`http_service.dart`)
- WebSocket re-auth using real JWT instead of mock token

### B. Biometric Unlock
- On app launch, if a refresh token is in secure storage, prompt for biometric (`local_auth` package)
- On success, decrypt + use refresh token to mint a new access token → user is logged in without re-entering credentials
- On biometric failure or unavailable hardware, fall back to password entry (email is pre-filled from last-used)
- No PIN fallback in Tier 1 (rely on device lock)

### C. WebSocket Session Persistence
- Cache the `session_id` (e.g. `"wise penguin"`) in secure storage, keyed by user email
- On reconnect, attempt to resume the cached session ID; only request a new one if the server rejects it
- Survive app restarts and network drops without losing the session

### D. Server Context Toggle (Dev ↔ Test)
- Add a `ServerContext` setting: `dev` (e.g., `http://localhost:7999`) or `test` (separate URL TBD)
- One context active at a time — switching contexts:
  - Logs out the current session
  - Clears the cached session ID and access token
  - Keeps separate per-context storage of refresh token + last-used email (so each context remembers its own login)
- Surfaced in Settings screen with a clear "Active server: DEV" / "Active server: TEST" indicator (probably color-coded for accident prevention)
- App startup honors the previously-selected context
- Configurable URLs (per-context base URL + WS URL) loaded from a bundled config file with optional override

### E. Cleanup of mock auth
- Remove `Bearer mock_token_email_{userId}` everywhere
- Replace with real `Bearer {access_token}`

## Decisions (resolved 2026-04-15)

| Question | Decision |
|----------|----------|
| Registration | **Defer** — login-only; users must pre-exist on backend |
| Email verification UX | **Defer** — no banner, no gating |
| Multi-account | **Single account at a time** |
| Biometric unlock | **Yes** — required when refresh token present; password fallback |
| Last-used email persistence | **Yes** — pre-fill login form |
| Dev test credentials | **Defer** — no env-var shortcut |
| Server context toggle | **New scope**: Dev ↔ Test, one at a time |

## Omitted from Tier 1 (track for later tiers)

These backend endpoints / features are intentionally postponed:

- `POST /auth/register` — registration UI
- `POST /auth/request-password-reset` + `POST /auth/reset-password` — password reset flow
- `POST /auth/request-verification` + `POST /auth/verify-email` — email verification flow
- `PUT  /auth/change-password` — change-password screen
- Multi-account switching
- PIN fallback for biometric
- Dev/test credential automation
- Production server context (only Dev + Test for now)

## Resolved Open Questions (2026-04-15)

| Question | Decision |
|----------|----------|
| Test server URL | Same host as Dev, port **8000** (Dev = 7999, Test = 8000) |
| Default context on first launch | **Dev** |
| Biometric fallback when no enrollment | **Password-only login** (no block) |
| "Remember me" checkbox | Defer — always store refresh token; biometric/password guards it |

8. Build/refresh login screen with email pre-fill
