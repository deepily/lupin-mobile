## Context

Backend exposes a **17-endpoint notification system** (senders → conversations → messages, with played markers, soft-delete by date, generate-gist) and a **9-endpoint decision-proxy/trust system** (trust modes, pending decisions, ratification, batch acknowledgement). Mobile has **zero REST integration** with either; `NotificationBloc` is a skeleton with no UI.

The data layer was hand-coded against the live OpenAPI schema + router source on 2026-04-16: 26 typed methods across two repositories, plus 18 model classes covering every documented response shape.

## Scope

### A. Notifications data layer (✅ done)
- `lib/features/notifications/data/notification_models.dart` — 18 models (NotificationItem, ConversationMessage, SenderSummary, DateSummary, ProjectSession, GistResponse, NotifyDispatchResponse, NotificationResponseAck, plus envelopes / requests).
- `lib/features/notifications/data/notification_repository.dart` — 17 typed methods over the shared Dio.

### B. Decision-proxy data layer (✅ done)
- `lib/features/decision_proxy/data/decision_proxy_models.dart` — `TrustMode` enum + `ProxyDecision`, `PendingSummary`, `TrustStateItem`, `RatifyResponse`, `AcknowledgeResponse`, `BatchIdResponse`, `TrustModeStatus`, `TrustModeUpdateRequest/Response`.
- `lib/features/decision_proxy/data/decision_proxy_repository.dart` — 9 typed methods.

### C. Notifications UI (next)
- **NotificationBloc** — events: `NotificationsLoadSenders`, `NotificationsLoadConversation`, `NotificationsMarkPlayed`, `NotificationsRespond`, `NotificationsBulkDelete`, `NotificationsRefresh`. States: loading / loaded(senders) / loadedConversation / responding / error.
- **InboxScreen** — multi-sender inbox grouped by `sender_id`, badge with `new_count`, sort by `last_activity` desc; pull-to-refresh; per-sender swipe-to-soft-delete.
- **ConversationScreen** — thread view per sender; date-grouped (uses `conversation-by-date`); chip showing `state` (pending/delivered/responded/expired); tap-to-open `InteractivePromptSheet` for messages where `response_requested == true`.
- **InteractivePromptSheet** — bottom sheet variants:
  - `yes_no` → two FilledButtons (Yes / No) with optional comment field
  - `multiple_choice` → radio list from `response_options` (single-select) or checkbox list (multi-select), free-form "Other" field
  - `open_ended` → multiline TextField
  - All submit via `NotificationRepository.respond(NotificationResponsePayload)`; on success, mark notification played
- **WebSocket integration** — listen for `notification_queue_update` event → re-fetch senders or append to current conversation if it matches `active_sender_id`.

### D. Decision-proxy UI (next)
- **DecisionProxyBloc** — events: `LoadMode`, `SetMode`, `LoadPending`, `RatifyDecision`, `DeleteDecision`, `LoadTrustState`, `AcknowledgeBatch`. States: idle / loading / modeLoaded / pendingLoaded / ratified / error.
- **TrustDashboardScreen**:
  - Header: current `TrustModeStatus` (`ini_mode`, `running_mode`, `effective`) — color-coded (disabled=grey, shadow=blue, suggest=orange, active=green); SegmentedButton to switch modes (PUT `/api/proxy/mode`); confirmation dialog when downshifting from `active`.
  - Body: pending decisions list (per-decision card showing question / category / trust_level / confidence + Approve/Reject IconButtons + optional feedback field + Delete swipe).
  - Footer: `total_pending` + `oldest_pending` summary; "Acknowledge batch" button calling `POST /api/proxy/acknowledge`.
- **TrustStateScreen** (drilldown) — per-domain trust levels with `total_decisions`, `successful/rejected` counts, `circuit_breaker_state`.

## Decisions (open — to confirm with user)

| Question | Default if not specified |
|----------|--------------------------|
| Push notifications (FCM/APNs)? | **Defer** to a later tier; in-app inbox + WS push only |
| Local notification mirror? | **Defer** — rely on in-app inbox |
| Voice-first interactive prompts (TTS the prompt, STT the answer)? | **Defer** — text/buttons only in Tier 2 |
| `generate-gist` UI? | Surface as "Summarize session" action on Conversation screen |
| Inbox refresh strategy? | Pull-to-refresh + WS-driven invalidation (no background polling) |
| Batch-id flow visibility? | Surface in TrustDashboard footer; auto-refresh `batch-id` after acknowledge |
| Trust mode default for new accounts | Use server-provided `effective` value; don't override |
