# [LUPIN-MOBILE] Re-Synchronizing Mobile App with Lupin API v0.1.6

## Context

We last touched `lupin-mobile` on **2025-07-10** (per README). It is now **2026-04-15** — about 9 months later. In the interim the parent Lupin backend has grown enormously: from a handful of endpoints (session + TTS + a single WebSocket) to **113 REST endpoints across 24 router groups** plus a dual-WebSocket dual-session architecture. The mobile app has scaffolding for many feature areas (auth, chat, notifications, queue, voice, etc.) but most of it is **stub-level only** — only TTS and session-id endpoints are actually wired.

This document is a **re-synchronization audit, not an implementation plan**. Its goal is to give the user a complete view of (a) every functional category the backend now exposes, (b) what the mobile app does today against each, and (c) the magnitude of the gap — so that the user can decide which categories to prioritize next.

## The Three Pillars the User Highlighted

### A. WebSocket
- ✅ Channels connect (`/ws/queue/{sid}`, `/ws/audio/{sid}`)
- ✅ 19 event types defined and dispatched
- 🟡 Subscription-based filtering manager *exists* but isn't routing per-user filters
- ❌ Session-ID persistence (the "wise penguin" `localStorage` model from web client) — not cached across app restarts
- ❌ No handling for the newer event families spawned by agentic jobs (notification queue updates with sender/conversation context, decision-proxy events)

### B. Notifications
- ❌ Zero REST integration with the 17-endpoint notification API
- ❌ No `Sender` / `Conversation` / `Gist` models
- ❌ No multi-sender inbox UI (the notifications feature dir has only a BLoC skeleton — no presentation layer)
- ❌ `/api/notify/response` (responding to interactive notifications) not implemented

### C. New backend functionality not yet categorized in the mobile app
The backend now offers entire **classes** of work the mobile app has no concept of:
1. **Real authentication** (JWT, refresh tokens, password reset, email verification)
2. **Decision proxy / trust ratification** (auto-approve flows, ratify pending decisions)
3. **Agentic long-running jobs** (deep research, podcast, presentation, SWE team) with their own lifecycle (submit → poll status → fetch artifacts via `/api/io/file`)
4. **Interactive Claude Agent SDK sessions** (`/api/claude-code/dispatch` + `inject`/`interrupt`/`end`) — bidirectional conversational agents
5. **Job lifecycle management** (cancel, retry, pause/resume, message-into-job, resume-from-checkpoint)
6. **Mode switching** (server-wide operating mode)
7. **User stats** (time-saved reporting)

## Recommended Next Step (to discuss with user)

This audit is **not** an implementation plan. The next conversation needs to decide: of the ~80 unimplemented endpoints, which categories matter for the mobile prototype's purpose? Suggested triage tiers to discuss:

- **Tier 1 (foundation, blocks everything else):** Real authentication (category 1) + WebSocket session persistence (category 4)
- **Tier 2 (user-visible value):** Notifications REST + multi-sender inbox UI (categories 5, 6) + Decision proxy handling (category 7)
- **Tier 3 (capability expansion):** Queue/CJ Flow REST + job lifecycle (category 8) + Interactive Claude Code sessions (category 17)
- **Tier 4 (specialized agentic UIs):** Deep research / podcast / presentation / SWE team (12–16) + IO file serving (23)

The user explicitly said: *"There's gonna be a ton of different so we need to talk further about what we're gonna do with this difference"* — so the immediate next move after approving this plan is a **scoping discussion**, not implementation.
