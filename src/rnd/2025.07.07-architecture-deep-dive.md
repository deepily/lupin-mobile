## BLoC Design Pattern Explained

### What is BLoC?

**BLoC (Business Logic Component)** is a reactive programming pattern that separates business logic from UI presentation. It was created by Google specifically for Flutter applications.

### Core Principles

#### 1. **Unidirectional Data Flow**
```
User Action → Event → BLoC → State → UI Update
```

#### 2. **Single Source of Truth**
Each piece of state has one authoritative source, managed by its corresponding BLoC.

#### 3. **Reactive Streams**
Uses Dart Streams for communication between layers, enabling real-time updates.

#### Step 3: Implement BLoC
```dart
class QueueBloc extends Bloc<QueueEvent, QueueState> {
  final WebSocketService _webSocketService;
  final QueueRepository _queueRepository;
  final NotificationService _notificationService;
  StreamSubscription<dynamic>? _webSocketSubscription;
  
  QueueBloc({
    required WebSocketService webSocketService,
    required QueueRepository queueRepository,
    required NotificationService notificationService,
  }) : _webSocketService = webSocketService,
       _queueRepository = queueRepository,
       _notificationService = notificationService,
       super(QueueInitial()) {
    
    // Register event handlers
    on<QueueStarted>(_onQueueStarted);
    on<QueueJobAdded>(_onQueueJobAdded);
    on<QueueNotificationReceived>(_onQueueNotificationReceived);
    on<QueueJobStatusChanged>(_onQueueJobStatusChanged);
  }
  
  Future<void> _onQueueStarted(
    QueueStarted event,
    Emitter<QueueState> emit,
  ) async {
    emit(QueueLoading());
    
    try {
      // Load initial data
      final jobs = await _queueRepository.getAllJobs();
      final notifications = await _notificationService.getNotifications();
      
      // Start WebSocket connection
      await _webSocketService.connect();
      
      // Listen to WebSocket messages
      _webSocketSubscription = _webSocketService.stream.listen(
        (message) => _handleWebSocketMessage(message),
      );
      
      emit(QueueLoaded(
        todoJobs: jobs.where((j) => j.status == JobStatus.todo).toList(),
        runningJobs: jobs.where((j) => j.status == JobStatus.running).toList(),
        completedJobs: jobs.where((j) => j.status == JobStatus.completed).toList(),
        deadJobs: jobs.where((j) => j.status == JobStatus.dead).toList(),
        notifications: notifications,
        isConnected: true,
      ));
    } catch (error) {
      emit(QueueError(error: error.toString()));
    }
  }
  
  void _handleWebSocketMessage(dynamic message) {
    switch (message['type']) {
      case 'job_update':
        final job = Job.fromJson(message['job']);
        add(QueueJobStatusChanged(
          jobId: job.id,
          newStatus: job.status,
        ));
        break;
      case 'notification':
        final notification = NotificationItem.fromJson(message['notification']);
        add(QueueNotificationReceived(notification: notification));
        break;
      case 'new_job':
        final job = Job.fromJson(message['job']);
        add(QueueJobAdded(job: job));
        break;
    }
  }
  
  Future<void> _onQueueNotificationReceived(
    QueueNotificationReceived event,
    Emitter<QueueState> emit,
  ) async {
    if (state is QueueLoaded) {
      final currentState = state as QueueLoaded;
      final updatedNotifications = [
        event.notification,
        ...currentState.notifications,
      ];
      
      emit(QueueLoaded(
        todoJobs: currentState.todoJobs,
        runningJobs: currentState.runningJobs,
        completedJobs: currentState.completedJobs,
        deadJobs: currentState.deadJobs,
        notifications: updatedNotifications,
        isConnected: currentState.isConnected,
      ));
      
      // Handle audio notification if needed
      if (event.notification.hasAudio && event.notification.audioText != null) {
        _notificationService.playAudioNotification(event.notification.audioText!);
      }
      
      // Show system notification
      _notificationService.showSystemNotification(event.notification);
    }
  }
}
```

#### Step 4: Use in UI
```dart
class QueueDashboardScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return BlocBuilder<QueueBloc, QueueState>(
      builder: (context, state) {
        if (state is QueueLoading) {
          return Center(child: CircularProgressIndicator());
        }
        
        if (state is QueueError) {
          return Center(child: Text('Error: ${state.error}'));
        }
        
        if (state is QueueLoaded) {
          return _buildDashboard(state);
        }
        
        return Center(child: Text('Initializing...'));
      },
    );
  }
  
  Widget _buildDashboard(QueueLoaded state) {
    return Column(
      children: [
        // Connection Status
        ConnectionStatusBar(isConnected: state.isConnected),
        
        // Notifications Badge
        NotificationBadge(
          count: state.notifications.where((n) => !n.isRead).length,
          onTap: () => _showNotifications(context, state.notifications),
        ),
        
        // Job Queue Tabs
        JobQueueTabs(
          todoJobs: state.todoJobs,
          runningJobs: state.runningJobs,
          completedJobs: state.completedJobs,
          deadJobs: state.deadJobs,
        ),
      ],
    );
  }
}
```

## Complete System Architecture

### 1. **Queue System - 5 Components (Including Notifications)**

```dart
enum QueueType {
  todo,           // Jobs waiting to be processed
  running,        // Jobs currently being executed
  completed,      // Successfully finished jobs
  dead,           // Failed jobs
  notifications,  // Ad-hoc communications and alerts
}
```

#### Job Queues (4 Traditional Queues)
- **Todo Queue**: Pending jobs waiting for processing
- **Running Queue**: Jobs currently being executed
- **Completed Queue**: Successfully finished jobs
- **Dead Queue**: Failed or cancelled jobs

#### Notifications Queue (5th Queue)
- **Purpose**: Handle all ad-hoc communications outside normal job flow
- **Types**: System alerts, user messages, error notifications, audio responses
- **Characteristics**: 
  - Real-time delivery via WebSocket
  - Audio capability for TTS announcements
  - Persistent storage for offline viewing
  - Rich metadata for context

### 2. **Notification System Architecture**

```dart
class NotificationService {
  final WebSocketService _webSocketService;
  final TTSService _ttsService;
  final LocalNotificationService _localNotificationService;
  final NotificationRepository _repository;
  
  // Handle incoming notifications from WebSocket
  void handleNotification(NotificationItem notification) {
    // Store locally
    _repository.saveNotification(notification);
    
    // Show system notification
    _localNotificationService.show(notification);
    
    // Play audio if enabled
    if (notification.hasAudio && notification.audioText != null) {
      _ttsService.speak(notification.audioText!);
    }
    
    // Update UI via BLoC
    _notificationBloc.add(NotificationReceived(notification));
  }
  
  // Send notification to backend
  Future<void> sendNotification(String message, NotificationType type) async {
    final notification = NotificationItem(
      id: uuid.v4(),
      title: _getTitleForType(type),
      message: message,
      type: type,
      timestamp: DateTime.now(),
    );
    
    _webSocketService.sendMessage({
      'type': 'send_notification',
      'notification': notification.toJson(),
    });
  }
}
```

### 6. **Notification Types and Handling**

```dart
enum NotificationType {
  // System Notifications
  info,              // General information
  warning,           // Warnings that need attention
  error,             // Error messages
  success,           // Success confirmations
  
  // Job-Related Notifications
  jobStarted,        // Job began processing
  jobCompleted,      // Job finished successfully
  jobFailed,         // Job encountered an error
  jobPaused,         // Job was paused
  jobResumed,        // Job was resumed
  
  // User Communications
  userMessage,       // Direct message to user
  systemAlert,       // System-wide alerts
  maintenanceNotice, // Scheduled maintenance
  
  // Audio/TTS Notifications
  audioResponse,     // TTS audio response
  voiceCommand,      // Voice command feedback
  audioAlert,        // Audio-only alerts
  
  // Real-time Updates
  liveUpdate,        // Real-time status updates
  dataSync,          // Data synchronization status
  connectionStatus,  // Connection state changes
}

class NotificationHandler {
  static void handleNotification(NotificationItem notification) {
    switch (notification.type) {
      case NotificationType.error:
        _showErrorDialog(notification);
        _logError(notification);
        break;
        
      case NotificationType.jobCompleted:
        _playSuccessSound();
        _updateJobStatus(notification);
        break;
        
      case NotificationType.audioResponse:
        _playTTSResponse(notification.audioText!);
        break;
        
      case NotificationType.userMessage:
        _showPersistentNotification(notification);
        break;
        
      default:
        _showStandardNotification(notification);
    }
  }
}
```

## Why This Architecture?

### 1. **Scalability**
- Each BLoC handles a specific domain
- Easy to add new features without affecting existing code
- Clear separation of concerns

### 2. **Testability**
- Business logic is isolated in BLoCs
- Easy to mock dependencies
- Comprehensive test coverage possible

### 3. **Maintainability**
- Predictable data flow
- Clear responsibility boundaries
- Easy to debug and trace issues

### 4. **Real-time Capability**
- Reactive streams handle real-time updates seamlessly
- WebSocket integration is clean and manageable
- State updates propagate automatically to UI

### 5. **Offline Support**
- State can be persisted locally
- Graceful handling of connectivity issues
- Queued actions can be replayed when online

This architecture ensures the mobile app can handle the complex real-time requirements of the Lupin system while maintaining code quality and developer productivity.
